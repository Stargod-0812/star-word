**Redis 集群升级到 7.x 的迁移方案**

## 背景与现状
当前线上 Redis 集群运行在 5.x/6.x 版本，已稳定支撑缓存、会话和部分异步任务场景。但现版本在命令能力、内存管理、复制链路稳定性和运维工具支持方面已逐步落后，后续安全修复和社区支持周期也有限。随着业务规模扩大，现网对高可用、可观测性和升级可维护性的要求持续提高，现有版本已成为基础设施演进的约束。

## 目标
本次迁移目标是将 Redis 集群升级到 7.x，在不改变业务访问方式和数据模型的前提下，提升集群稳定性、性能与后续可维护性。升级后需满足以下条件：客户端协议兼容，核心读写链路无功能回归，主从切换与故障恢复机制正常，监控、告警与备份流程保持可用，整体业务中断时间控制在可接受范围内。

## 迁移步骤
第一阶段，在测试环境搭建 Redis 7.x 集群，完成业务压测、命令兼容性检查、Lua 脚本与客户端 SDK 验证。  
第二阶段，在生产环境新增一套 7.x 集群，按现网拓扑配置主从、副本、持久化和监控。  
第三阶段，通过离线全量同步加增量校验完成数据迁移，并对热点 key、过期策略和容量水位进行核对。  
第四阶段，选择低峰时段，按业务分批切换流量，先灰度一小部分实例，观察延迟、错误率、命中率和主从复制状态，再逐步全量切换。  
第五阶段，切换完成后持续观察 24 小时，确认运行稳定，再下线旧集群。

## 回滚预案
若灰度或全量切换后出现大面积超时、命令不兼容、数据异常或复制故障，立即停止扩容切换，业务流量回指旧集群。回滚期间保留旧集群只读或双写能力，确保缓存可快速恢复。若新集群已写入关键数据，则按预先定义的数据校验与补偿流程处理。回滚完成后复盘根因，修复后再重新安排迁移窗口。
