# RFC：Redis 集群升级到 7.x 的迁移方案

## 现状

当前生产集群运行在 Redis 5.x，已稳定承载缓存、分布式锁与部分延迟队列请求。现网节点数、分片方式与主从拓扑已经固定，但版本偏旧，存在两个问题：一是新版本缺陷修复与性能改进无法获得；二是后续客户端能力、运维工具链与安全补丁会逐步向 7.x 靠拢，继续停留在旧版本，维护成本只会升高。

## 目标

本次迁移目标是把生产集群升级到 Redis 7.x，保持拓扑不变、键空间不变、业务接口不变。迁移期间单次写抖动控制在秒级，业务侧无需改代码，只需配合发布窗口观察。升级完成后，集群应满足现网容量、延迟与主从切换要求。

## 迁移步骤

采用“新集群并行、灰度切流、稳定后下线旧集群”方案，不在原地滚动升级。先按现网规格创建 7.x 集群，导入参数配置，逐项核对持久化、淘汰策略、慢日志阈值与 ACL 设置。随后从旧集群全量同步数据，再补齐增量写入。校验键数量、过期时间分布、热点键命中率与核心业务读写结果。校验通过后，先让低风险业务切到新集群，观察一小时；无异常，再按业务分组扩大流量，直到旧集群只保留只读观察流量。全部稳定后，冻结旧集群写入，完成最终切换。

## 回滚预案

回滚条件设为三类：核心接口错误率上升、P99 延迟超阈值、数据校验失败。触发任一条件，立刻停止后续切流，把流量切回旧集群，并保留新集群现场用于排障。回滚期间不对旧集群做结构变更，确保其始终可接管全部请求。故障原因查清前，不再次发起迁移。
