# 缓存一致性方案：为什么我们选了 Cache-Aside

我们团队最近为订单服务选了 Cache-Aside Pattern 做缓存一致性。这篇说一下选的理由，以及为什么没选另外三种方案。

## 场景

订单服务：日均 500w 读、30w 写，读写比 ~17:1。用的 MySQL 5.7 + Redis 6。一致性要求：90 秒内最终一致可接受。

## 四种方案，我们都评估了

| 方案 | 一致性窗口 | 实现复杂度 | 高并发下的缺陷 |
| --- | --- | --- | --- |
| 先更新库，再更新缓存 | 窗口大 | 低 | 并发写会产生脏缓存，两次写交叉后缓存拿到旧值 |
| 先更新缓存，再更新库 | 窗口大 | 低 | 库写失败时缓存是脏的，不可接受 |
| 先删缓存，再更新库 | 窗口中 | 中 | 读写交叉会把旧值读回缓存 |
| 先更新库，再删缓存（Cache-Aside） | 窗口小 | 中 | 删缓存失败时要重试；少量脏读可接受 |

## 我们为什么选 Cache-Aside

三个原因，按优先级：

1. **读多写少的场景里，删缓存比更新缓存更合适**。我们的读写比 17:1，每次写去更新缓存是浪费 —— 下一次读自然会重新加载。
2. **应对删缓存失败有成熟兜底**。用消息队列做补偿重试，保证最终删除。实测缓存脏数据存在时长 P99 < 2 秒，满足 90 秒窗口。
3. **Redis + MySQL 组合已被大量生产环境验证**。Spring Cache、Canal + MQ 方案齐全，团队踩过的坑前人都踩过。

## 为什么不选其他三种

- **先更新缓存再更新库**：直接否决。库写失败会留下脏缓存，订单数据不能脏。
- **先更新库再更新缓存**：实现简单，但并发写时容易出现"A 写库 → B 写库 → B 写缓存 → A 写缓存"导致缓存落后。我们读 QPS 高，错一次影响范围大。
- **先删缓存再更新库**：用 delay-double-delete 能解，但多一次延迟删带来额外链路复杂度，收益不大。

## 补偿链路

删缓存失败的兜底：Canal 订阅 binlog，发到 Kafka 的 `cache-invalidation` topic，独立消费者兜底删除。Canal 消费延迟目前 P99 < 1 秒。

## 风险

- 短时间脏读不可消除。业务侧接受。
- Canal 或 Kafka 宕机时，补偿链路断。监控告警已覆盖，SLA 99.9% 有冗余。
